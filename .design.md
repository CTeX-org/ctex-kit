# mac15plus fontset 设计讨论

Issue: #722 | PR: #782 | 分支: `fix/722-mac15plus-fontset`

## 问题根因

macOS 15 (Sequoia) 起，`/System/Library/Fonts/PingFang.ttc` 不再存在，导致 ctex 的 `fontset=mac` 自动检测逻辑（基于该文件是否存在判断 macnew/macold）一律 fallback 到 `macold`。macold 所依赖的 STKaiti/STHeiti/STSong 等字体在新系统上也不一定可用，尤其 LuaLaTeX (freetype/luaotfload) 基本全部失败。

## macOS 26.4.1 实测字体可用性矩阵

| 字体 | fontconfig | luaotfload (freetype) | XeLaTeX (Core Text) |
|------|:---------:|:--------------------:|:-------------------:|
| Songti SC (Light/Regular/Bold/Black) | YES | YES | YES |
| STSong | YES | YES | YES |
| Heiti SC (Light/Medium) | YES | YES | YES |
| STHeiti | YES | YES | YES |
| PingFang SC | NO | NO | NO |
| Kaiti SC | NO | NO | YES |
| STKaiti | NO | NO | YES |
| STFangsong | NO | NO | YES |
| Baoli SC | NO | NO | YES |
| Yuanti SC (Light/Regular/Bold) | NO | NO | YES |

关键差异：XeLaTeX 通过 Core Text API 查找字体，能访问系统字体注册表中的大部分字体。LuaLaTeX 走 luaotfload (freetype + fontconfig)，无法发现被 Apple 移至私有框架目录的字体。PingFang 因格式变更 (`hvgl` table) 三个后端均无法使用。

## macOS 15.7.4 实测（muzimuzhi）

与 macOS 26 存在差异：

- PingFang.ttc 作为 downloadable 字体仍存在（在 AssetsV2 中），XeLaTeX 直接可用
- LuaLaTeX 设置 OSFONTDIR 后也可使用 PingFang
- Kaiti SC、Baoli SC、STFangsong、Yuanti SC 等也是 downloadable 字体
- macOS 15 上 PingFang 并未"变成系统 UI 专用"——PingFang.ttc 与 PingFangUI.ttc 并存

## 当前 PR #782 方案

- 新增 `fontset=mac15plus`
- `fontset=mac` 三级自动检测：mac15plus -> macnew -> macold
- 无衬线族统一使用 Heiti SC（PingFang 不可访问）
- LuaTeX 通过 Lua `lfs` 扫描 AssetsV2 目录定位字体文件并显式加载
- 不定义 `\pingfang` 命令，`\yahei` 映射为 `\heiti`
- pdfTeX/upTeX 下 mac15plus 报错（不支持）
- 已修复 Lua 侧 `\l__ctex_mac_stfangso_dir_tl` 未定义 bug (2b2b75f)
- Harry-Chen 在 macOS 26.3.1 测试通过

## muzimuzhi 替代方案（2026-04-26）

### 建议 1：退役 macold

macold 是为 OS X El Capitan (10.11, 2015 年) 之前准备的。建议不再自动选择 macold。

### 建议 2：不新增 fontset，在 macnew 内动态判断

> 甚至可以不提供新的 fontset，在 macnew 里做判断，
> `\fontspec_font_if_exist:nTF { PingFang SC }` 为真时仍使用 PingFang SC
> （并定义 `\pingfang`），否则改用 Heiti SC。

即 `fontset=mac` 简化为 macnew -> macold（甚至只 macnew），在 macnew 内部按字体存在性动态决定。

## 方案对比

| 维度 | PR #782 当前方案 | muzimuzhi 方案 |
|------|-----------------|---------------|
| fontset 数量 | 新增 mac15plus | 不新增，复用 macnew |
| 检测层级 | mac15plus -> macnew -> macold | macnew -> macold（或只 macnew） |
| PingFang 处理 | 一律不用 | `\fontspec_font_if_exist:nTF` 动态判断 |
| macOS 15 兼容 | 统一走 mac15plus（可能丢掉仍可用的 PingFang） | macnew 内动态判断，能用就用 |
| LuaTeX 特殊处理 | Lua lfs 扫描 AssetsV2 + 显式 Path 加载 | 如 PingFang 不存在则回退 Heiti SC |
| 维护成本 | 新增一个 .def 文件 + Lua 扫描逻辑 | 改动集中在 macnew 内部 |
| 用户心智模型 | mac/macnew/macold/mac15plus 四选一 | mac/macnew/macold 三选一（macold 可能退役） |

## 风险点

1. **downloadable 字体的下载延迟**：macOS 的 downloadable 字体在首次使用时可能触发弹窗下载。`\fontspec_font_if_exist:nTF` 的运行时检测是否会受此影响？
2. **macold 退役影响**：仍在 OS X 10.10 及更早系统上的极少数用户需要手动指定 `fontset=macold`
3. **macOS 15 vs 26 差异**：PingFang 在 macOS 15 上可用（downloadable），在 macOS 26 上不可用（格式变更 hvgl table）。是否需要区分？
4. **CI 测试限制**：GitHub Actions macos-latest 映射到 macos-15，无法覆盖 macOS 26 环境；且 CI runner 上字体的存在性可能与真实用户环境不同
5. **AssetsV2 路径稳定性**：Apple 有频繁修改系统字体路径的历史（参考知乎文章 [1](https://zhuanlan.zhihu.com/p/42434849) 和 [2](https://zhuanlan.zhihu.com/p/29483442714)）

## Apple CJK 字体变更历史

### 官方字体列表（Apple Support 文档）

Apple 为每个 macOS 大版本发布字体列表页面（如 [macOS Sequoia: support.apple.com/120414](https://support.apple.com/120414)）。对比 macOS Monterey (12)、Ventura (13)、Sonoma (14)、Sequoia (15) 的字体列表，CJK 字体的**名单本身几乎无变化**——PingFang SC、Songti SC、Heiti SC、Kaiti SC、STSong、STHeiti、STKaiti、STFangsong、Baoli SC、Yuanti SC 等在每一版中均列为"Installed or downloadable"。

Apple 的官方文档**从不区分**哪些字体是预装的、哪些是需要按需下载的。统一说法是：
> "Most of these fonts are installed and enabled automatically. Others in this list can be downloaded using Font Book."

**结论：Apple 官方文档未记录 macOS 15 的字体路径变更或格式变更。** 这些变化是未文档化的系统内部实现细节。

### macOS 字体路径变迁史（社区整理 + 实测）

| macOS 版本 | PingFang.ttc 位置 | 格式 | 备注 |
|-----------|-------------------|------|------|
| 10.11 El Capitan (2015) ~ 14 Sonoma | `/System/Library/Fonts/PingFang.ttc` | 传统 TrueType Collection | ctex `macnew` 判断依据 |
| 15 Sequoia (2024) | `/System/Library/AssetsV2/com_apple_MobileAsset_Font7/<hash>.asset/AssetData/PingFang.ttc` | 传统格式（可被 freetype 解析） | PingFang 变为 downloadable 字体，路径含随机 hash；XeLaTeX 通过 Core Text 可用，LuaLaTeX 需设 OSFONTDIR |
| 15 Sequoia (2024) | `/System/Library/PrivateFrameworks/FontServices.framework/Resources/Reserved/PingFangUI.ttc` | **新格式 (hvgl table)** | Apple 私有 HVF 格式，freetype/fontconfig 无法解析 |
| 26 Tahoe (2026) | 同 15，但 PingFang.ttc 可能不存在（取决于是否已下载） | 混合 | macOS 26 上实测：PingFang SC 三个后端均不可用 |

### `hvgl` 表——Apple 私有字体格式

macOS 15.6 / iOS 18.6 起，Apple 引入了 **HVF (Hierarchical Variation Font)** 格式，通过 `hvgl` 表定义字形。

技术要点（来源：[Apple TrueType Reference](https://developer.apple.com/fonts/TrueType-Reference-Manual/RM06/Chap6hvgl.html)）：

- **层次化字形定义**：字形由 composites（组合件）和 shapes（形状）构成树形结构
- **32 位字形索引**：传统 sfnt 字体表仅支持 16 位字形编号，现有字体渲染引擎无法寻址 hvgl 字形
- **小端序 + IEEE 754 浮点**：与传统 TrueType 的大端序整数坐标完全不同
- **稀疏 CSC 矩阵存储变体数据**：用于高效存储可变字体的轴插值数据
- **Apple 专有**：仅 macOS 15.6+/iOS 18.6+ 原生支持，无跨平台标准化

**PingFangUI.ttc** 使用了此格式。这意味着：
- `otfinfo` 报 "bad magic number"
- `fc-scan` 无输出
- FreeType 无法解析
- XeTeX 的 `[/path/PingFangUI.ttc]` 语法（依赖 FreeType）也无法加载
- 即**无论按名称还是按文件路径，XeTeX 和 LuaTeX 均无法加载 PingFangUI.ttc 中的字体**

### 传统 PingFang.ttc vs PingFangUI.ttc

在 macOS 15 上两者并存：
- `PingFang.ttc`（downloadable，传统格式，在 AssetsV2 随机 hash 目录中）——XeLaTeX 可用，LuaLaTeX 设 OSFONTDIR 后可用
- `PingFangUI.ttc`（系统内置，hvgl 格式，在 PrivateFrameworks 中）——所有第三方引擎不可用

muzimuzhi 在 macOS 15.7.4 上确认：下载 PingFang.ttc 后，`\fontspec{PingFang SC}` 在 XeLaTeX 直接可用、LuaLaTeX 设 OSFONTDIR 后也可用。

### 其他 CJK 字体的 downloadable 状态

muzimuzhi 在 macOS 15.7.4 Font Book 中确认以下字体变为 downloadable：
- Baoli SC
- Kaiti SC
- PingFang SC
- STFangsong
- Yuanti SC

macOS 的 downloadable 字体在应用使用时可能触发系统弹窗提示下载（PingFang 除外——macOS 15 上 PingFang 不会触发自动下载弹窗）。macOS 26 上，Liam 实测部分 downloadable 字体（如 Baoli）会在 TeX 编译时触发自动下载。

### LuaLaTeX 不可发现的字体分析

LuaLaTeX 找不到的 3 个字体（PingFang SC、Baoli SC、Yuanti SC）有一个共同特征：它们的文件都在 **AssetsV2** 目录中。luaotfload 的字体搜索路径不包含 AssetsV2，而 OSFONTDIR 环境变量未设置。

fontconfig (`fc-match`) 能发现这些字体，说明 fontconfig 的搜索范围比 luaotfload 更广（可能通过 Core Text 获取字体列表后回写缓存）。

### fontset=macnew 编译测试

- XeLaTeX + macnew: **SUCCESS**
- LuaLaTeX + macnew: **FAILED** (`"PingFang SC" cannot be found`)

### fontset=mac 自动检测

当前 CTAN 版本 (v2.5.10)：`fontset=mac` → 检测 `/System/Library/Fonts/PingFang.ttc` 不存在 → **fallback 到 macold**。

### 关键结论

1. **macOS 15 上 PingFang.ttc（传统格式）完全可用于 XeLaTeX**——文件存在于 AssetsV2，fontconfig 可发现，Core Text 可加载。PR #782 文档说的"苹方已改为系统 UI 专用"在 macOS 15 上**不成立**。
2. **LuaLaTeX 不可用的原因不是字体格式问题，而是搜索路径问题**——luaotfload 不搜索 AssetsV2。这是 luaotfload 的配置/适配缺陷，不是 PingFang 本身不可用。
3. **LuaLaTeX 找不到的字体 = 文件在 AssetsV2 中的字体**——PingFang SC、Baoli SC、Yuanti SC 均如此。而 Kaiti SC、STFangsong 等在 `/Library/Fonts/` 中的字体两个引擎都能找到。
4. **fontset=macnew 在 XeLaTeX 下完全正常**，LuaLaTeX 下因 PingFang 搜索路径问题失败。

### 与之前结论的对比修正

## AssetsV2 路径结构分析

### 路径中的 hash 是否随机？

**不是随机的，是基于 asset 版本的确定性 hash。**

- 本机 (macOS 15.7.3) PingFang hash: `3419f2a427639ad8c8e139149a287865a90fa17e`
- muzimuzhi (macOS 15.7.4) PingFang hash: `3419f2a427639ad8c8e139149a287865a90fa17e`
- hash ≠ 文件内容 SHA-1（已验证），而是 Apple MobileAsset 系统生成的 asset 标识符
- 同版本 asset 在不同机器上 hash 一致；**但 asset 更新时 hash 会变**，因此不能硬编码

### 完整路径结构

macOS 15 上的 AssetsV2 字体完整路径模式：
```
/System/Library/AssetsV2/com_apple_MobileAsset_Font7/<hash>.asset/AssetData/<FontFile>
```

### ctex 层面是否能处理？

#### XeLaTeX：不需要处理

XeLaTeX 通过 Core Text 按字体名查找，AssetsV2 中的 downloadable 字体自动可发现。

#### LuaLaTeX：两种方案

**方案 A — Lua `lfs` 遍历 AssetsV2（PR #782 现有思路）**

- 可行：`lfs.dir()` 遍历 `*.asset/AssetData/` 匹配文件名，不硬编码 hash
- 已验证：能找到 PingFang.ttc、Baoli.ttc 等所有字体
- 找到后通过 fontspec 的 `Path =` 语法加载，已验证 LuaLaTeX 下成功
- 风险：依赖 `com_apple_MobileAsset_Font7` 目录名不变

**方案 B — `\fontspec_font_if_exist:nTF` 动态回退（muzimuzhi 方案）**

- 不关心路径，按字体名检测
- XeLaTeX：PingFang SC 存在 → 用 PingFang
- LuaLaTeX：PingFang SC 不存在 → 回退 Heiti SC
- 优点：不依赖任何路径假设
- 缺点：LuaLaTeX 在 macOS 15 上永远用不了 PingFang（即使文件存在）

### macOS 版本间 AssetsV2 目录名差异

macOS 15 上的目录名为 `com_apple_MobileAsset_Font7`。
根据 #722 早期讨论，macOS 26 上可能已变为 `Font8` 或其他编号。
如果采用方案 A（lfs 遍历），需要处理目录名的版本差异。
