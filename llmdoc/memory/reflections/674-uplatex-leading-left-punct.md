# 674: upLaTeX 行首左标点压缩限制分析

## 现象

issue #674 关注 upLaTeX 下行开头左标点未被压缩的问题。实测表明：段首位置在特定条件下可修复，但 `\\` 等显式换行后的新行行首左标点压缩仍然失效。

## 根因

根因不在 ctex 自己的宏逻辑，而在 upTeX JFM glue 机制的结构性限制。

调查确认 `upzhserif-h.tfm` 中把 TYPE 0 → TYPE 1（开括号）定义为 0.5em 的 shrinkable glue。问题在于：当 indent box 或其他边界对象处在行首位置时，JFM 会把它视为边界元素并触发这段 glue；而在宽松行中，这段 glue 不一定会被压缩掉，因此左标点看起来没有实现预期的“行首压缩”。

## 影响边界

### 可由宏层缓解的部分

段首场景可以通过 `\everypar + \inhibitglue` 抑制首个 JFM glue，从而改善段首左标点表现。这说明“新段落开始”仍有一定宏层介入空间。

### 超出 ctex 控制的部分

`\\` 后新行行首的问题本质上发生在 JFM / 引擎断行层。此时并不存在和段首等价的、稳定可用的宏层入口来消除对应 glue，因此需要 JFM 或引擎层面改动，超出了 ctex 可以单独修复的范围。

## 经验

- **先区分段首与普通换行行首**：两者表面现象相似，但可干预层级不同。
- **upLaTeX 排版异常往往要回到 JFM 看 TYPE 转移**：只盯 ctex 宏很容易误判为选项或 patch 问题。
- **边界盒会参与 JFM 配对**：indent box 不是“透明”的，它会改变 glue 插入时机。
- **要明确说明不可修复边界**：当问题落在 JFM/引擎层时，应把 ctex 能做与不能做的部分拆开写清楚，避免继续在宏层做无效 patch。
