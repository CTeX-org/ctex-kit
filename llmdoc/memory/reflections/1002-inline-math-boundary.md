# [Task Reflection]

## Task

在 #999 建立的 capture/register 框架上增加行内公式边界能力，使已注册命令包装的公式与去掉外层命令后的直接公式具有相同的左右间距。PR #1009 使用 `Closes #1002`，合并后关闭 #1002；对 #992 只作关联。实现同时更新用户手册、回归测试、外部可视矩阵和 PR 说明。

直接输入基准（oracle）始终保留公式形式。例如 `中\foo{$x$}文` 应与 `中$x$文` 比较，不能把基准换成西文字母 `x`。测试还要分别覆盖 `xCJKecglue=false/true`、默认／可区分间距和 `00/10/01/11` 四种源码空格。

## Expected vs Actual

- 预期：给 capture/register 增加一个 `math` 类别，再把 `$...$`、`\(...\)` 和 `\ensuremath{...}` 接入现有恢复出口，便能覆盖公式边界。
- 实际：公式语法出现在参数末尾，并不证明公式进入了可见输出。未知宏可能消费末尾分组、`$` 或 `\)`，所以源码扫描只能登记候选；必须等正文实际排完，再由列表末尾的 math 节点或 xeCJK 边界标记（marker）确认。
- 预期：公式后有一枚参数内源码空格时，可以用同一种“公式后有空格”状态处理。
- 实际：普通 stream（内容直接留在当前列表）中的空格仍在外层列表，会参与外层断行；box、wrapped-box（内容装入盒子）和 ulem 装饰流中的空格已经冻结在内部结构中，不再参与外层断行。两者需要分别表示为 `math-space` 和 `math-space-frozen`，补偿间距的计算方法也不同。
- 预期：宽度矩阵全绿就能证明间距恢复正确。
- 实际：总宽度可能掩盖左右两侧一多一少，单看自然宽度也看不出可伸缩间距（glue）的伸长量和收缩量是否可供段落装箱使用。最终验证必须同时包含左右边界比较、节点日志和有限容差下的实际断行。
- 预期：公式 marker 一经确认，后续透明或 post-transparent 命令仍可照常重放。
- 实际：颜色 special 节点或零尺寸盒子如果插入真实空格和 marker 之间，就破坏了 marker 成立所需的物理相邻关系。此时直接公式基准也不会跨越该节点恢复间距，marker 应当过期，而不是被移动到不可见节点之后。

## What Went Wrong

1. **把末尾公式语法过早当成可见输出。** 参数扫描看见 `{$x$}`、末尾 `$` 或末尾 `\)` 时，早期方案容易直接发布 `math` 末类别。但未知宏可能把这些记号当作普通参数或分隔参数的一部分，最终只排出 CJK 内容。源码形状只能说明“可能以公式结束”，不能代替实际节点证据。

2. **最初把所有参数内尾随空格看成同一种空格。** stream 中的真实空格仍在外层水平列表，其 stretch/shrink 已经参与外层断行；盒子与 ulem 中的空格则已被装入内部结构。若统一补足完整 `CJKecglue`，stream 会重复计算弹性；若统一扣除已有空格，冻结路径又会少掉外层可用的弹性。

3. **只检查自然宽度会产生假通过。** 两枚 glue 可以自然宽度相同，却具有不同的伸长量和收缩量。普通盒宽比较只能看到 natural width，无法证明段落在空间不足时真的能使用 `CJKecglue` 的 shrink。必须给 glue 设置可区分的弹性，并用有限段宽检查实际 badness 和段落高度。

4. **没有先把 `math-space` 的有效性写成物理相邻条件。** transparent（透明）颜色命令留下的 special 节点、post-transparent（后置透明）`\null` 留下的零尺寸盒子，都会隔开真实参数空格与 marker。若仍重放 marker，补偿 glue 会被错误地放到不可见节点之后，与直接输入的节点顺序不同。

5. **post-transparent 的一般恢复顺序不能直接套到过期的 `math-space`。** `\textnormal{$x$ }\hskip7pt\null` 在探测 marker 时需要暂存显式 7pt glue。确认 `math-space` 已过期后，必须先恢复这枚 glue，再恢复零尺寸盒子，保持“真实空格、显式 glue、`\null`”的顺序；若沿用一般 marker 的“盒子、marker、glue”顺序，就会把显式 glue 移到盒子之后。

6. **只比较整段总宽度不足以检查左右边界。** 左侧多出的间距可能恰好抵消右侧缺少的间距，最终总宽度仍相等。测试必须让公式分别处在命令开头和结尾，并独立构造左右边界 oracle。

7. **未合并 PR 的预览状态容易与 #992 的长期状态混淆。** PR 分支上的四套可视矩阵可以全部变绿，但 #992 的活表代表已经合并到主线的行为。若在 PR 合并前更新活表，就会把候选实现误写成主线事实，也失去从最终合并提交复验的依据。

8. **正式代码审查（code review）的独立性不能只靠“再看一遍代码”，也不能只靠“换一个子代理”。** 主代理既参与设计又协调实现，自检仍带有同一上下文和假设，不能兼任正式审查者。`bb14d1a3..2092edad` 后来由独立子代理审查，结论为 `APPROVE`，阻塞、重要和小问题均为 0；但报告没有记录子代理是否继承主对话，因此只能证明审查者身份独立，不能证明上下文隔离。llmdoc 增量审查据此把门禁收紧为新会话、受限输入和报告留痕三个条件。

9. **纯文字 PR 说明不能让维护者快速确认实际修复范围。** 公式边界同时受包装方式、两侧文字、源码空格和 `xCJKecglue` 影响，仅列测试数量仍很抽象。PR body 最终加入修复前／修复后的并排图片，以及“语法候选—实际节点确认—统一出口恢复”的流程图，才把问题、机制和结果连在一起。

## Root Cause

实现层的根因是混淆了三种不同事实：源码中出现了什么记号、命令实际排出了什么节点，以及这些节点是否仍处在能证明边界语义的物理位置。公式不会像普通字符那样自然触发完整的 interchar 类别转换，因此框架需要先从源码取得候选，再由实际节点确认；确认后的 marker 也只有在所依赖的真实空格仍与它相邻时才有效。

尾随空格问题还跨越了 TeX 的列表边界。stream 保留外层可伸缩 glue，box 和 ulem 则把空格冻结在内部。只保存自然宽度或只保存一个抽象的“有空格”标志，都会丢失后续断行需要的信息。

验证与协作层的根因是没有一开始就把证据的职责分开：总宽度、单侧边界、节点顺序、段落断行、可视预览、已合并状态和独立审查分别回答不同问题，任何一项都不能替代其余各项。

## What Worked

- 继续复用 #999 的 capture/register 分层，只给框架增加 `math`、`math-space` 和 `math-space-frozen` 能力，没有为字体、颜色、盒子、链接或 ulem 分别复制恢复算法。
- 所有尾部公式写法都采用“两阶段判断”：源码扫描只登记候选，正文排完后再检查真实 math 节点或 marker。消费可选参数、普通参数、分隔参数、`$` 和 `\)` 的反例证明这项确认确实阻止了误报。
- 普通 stream 用两对零净宽 kern 保存实际空格的伸长量和收缩量；计算外层补偿时扣除这枚真实空格已经提供的弹性。冻结路径完整保留 `CJKecglue` 的弹性，并把自然宽度差限制为不小于 0pt，避免较窄的 `CJKecglue` 把后续文字拉进框线或装饰范围。
- `command-boundary-math05` 不只记录盒宽，还把 direct、box、wrapped-box、stream 和 stream-ulem 的段宽缩短 1pt；五条路径的 badness 都是 12，证明 2pt shrink 实际参与了段落装箱。
- 对不可见节点采用与直接输入相同的相邻规则。颜色 special、直接 `\null` 和“`math-space`＋显式 7pt glue＋`\null`”分别固定末节点类型、宽度差、段落高度差和恢复顺序，没有把一般 post-transparent 算法扩大成任意向后扫描。
- 公式矩阵分别检查左右边界，并在默认／可区分间距与 `xCJKecglue=false/true` 四种配置下完成 5504 次比较；普通命令边界的 1668 个断言、xeCJK 109／109 和 ctex 184／184 也全部通过。
- PR body 使用修复前后的 `\fbox{$x$}` 图片展示肉眼可见的差异，再用流程图说明为何“识别语法”之后还要“确认实际节点”。图像负责展示结果，流程图负责解释因果，测试清单负责给出覆盖范围，三者没有互相替代。
- `bb14d1a3..2092edad` 的正式审查由独立子代理完成，并固定 base、head 和验证范围；这避免了主代理把自检包装成独立 code review。后续 llmdoc 增量审查又识别出报告未记录隔离方式的证据缺口。本轮最终审查将按收紧后的定义启动不继承主对话的新子代理，只提供允许输入，并在报告中记录隔离方式与固定范围。

## Promotion Candidates

- **语法观察只能产生候选，语义发布必须由实际输出确认。** 当宏参数可能被未知命令消费时，不应根据源码末尾形状直接决定可见输出类别。
- **跨列表恢复必须区分仍参与外层排版的状态和已经冻结在内部结构中的状态。** 对 glue 来说，这不仅包括自然宽度，还包括伸长量、收缩量和可用的断行位置。
- **marker 的含义可以依赖物理相邻关系。** 不可见节点一旦破坏该关系，就应按同条件直接输入的行为让 marker 过期；不能因为节点不可见就任意跨过它。
- **边界测试必须拆分左右两侧，并用宽度、节点和实际断行三层证据。** 总宽度只适合做第一层筛查，不能单独证明边界恢复正确。
- **预览结果与主线事实分开传播。** 未合并 PR 的图片和矩阵用于审查；issue 活表和规范图片必须以合并提交的复验结果为准。
- **实现者的自检与正式 code review 分开。** 正式审查使用不继承主对话的新子代理，只接收公开目标、完整 base/head SHA、范围和必要公开证据，并在报告中记录隔离方式与允许输入；任何后续代码变化都应重新做相应的隔离增量审查。
- **复杂 PR 要用图片展示结果、用流程图解释机制。** 测试数量不能代替维护者对修复前后行为和状态流转的理解。
