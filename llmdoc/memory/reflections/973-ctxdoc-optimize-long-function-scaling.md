# Reflection: PR 优化超长函数压缩机制及测宽时序修正

## 背景

在处理超长函数名时，`support/ctxdoc.cls` 原有的五档阶梯性均匀压缩机制在极端情况下（如压缩至 33% 以下）会导致 PDF 文本超出人眼分辨率阈值，严重影响阅读体验。同时，由于频繁的阶梯压缩循环和不精确的星号宽度估计，导致编译开销大且存在排版误差。

本次优化的目标是：限制过度均匀压缩、引入自适应放大机制、利用盒子精确计算符号宽度，并在经历本地深度复核后，修正测宽时序、统一变量作用域命名契约及文档规约。

## 核心发现与问题剖析

### 1. 超长压缩的体验阈值与望远镜乘积精确计算

实测表明，在 24" iMac Retina Display 100% 放大率下，函数名压缩至 50% 时阅读者已产生些许不适，压缩至 33% 及以下时则完全无法辨识。

**优化方案**：将均匀压缩的循环条件由 `> 1` 收窄为 `> 3`（初始值为 6），使得阶梯均匀压缩只执行 3 轮（6→5, 5→4, 4→3）。
- 其背后的缩放比例形成**望远镜乘积**：
  $$\frac{5}{6} \times \frac{4}{5} \times \frac{3}{4} = \frac{1}{2}$$
- 这确保了最低压缩下界精确锁定在原宽的 **50%**。超出 3 档后转为自适应机制，避免盲目浪费左侧空间，尽可能保证字体最大可读性。

### 2. 阻塞性问题：rEXP 盒子的变量前缀与作用域冲突

在代码重构过程中，出现了违反 expl3 命名契约的严重瑕疵：
- **表现**：EXP 盒子已完整改为全局变量 `\g__ctxdoc_function_EXP_box` 并配合 `\hbox_gset:Nn`；但 rEXP 盒子在声明和读取处仍为局域变量 `\l__ctxdoc_function_rEXP_box`，同时却错误地使用了全局赋值宏 `\hbox_gset:Nn`。
- **后果**：这种混用违反了 expl3 的 `l__` / `g__` 作用域命名契约，会破坏代码的健壮性与可读性。必须统一将声明、赋值和读取三处修正为全局前缀 `\g__ctxdoc_function_rEXP_box`。

### 3. 重要问题：数学字体延迟激活导致的“假精确”测宽陷阱

前期 Bot 审查曾盲目乐观地认为“类加载期执行盒子预加载属于时序安全”，该结论已被实测数据彻底证伪。

- **根本成因**：虽然 `\setmathfont` 在导言区调用，但 `unicode-math` 宏包实际上要等到 `\begin{document}` 阶段才会真正激活文档所需的数学字体（如 `texgyrepagella-math.otf`）。若在类加载期立即排版 `$\star$`，保存的其实是 **LaTeX2e 默认数学字体** 下的宽度，而非文档期的真实宽度。
- **实测数据对比**：

| 符号 | 字号 | 类加载期保存值（错误） | 文档期真实值（正确） | 偏差值 |
| :--- | :---: | :---: | :---: | :---: |
| `$\star$`（EXP） | 10pt | 5.00002pt | 5.85pt | ~0.85pt |
| `$\star$`（EXP） | 12pt | 5.87494pt | 7.02pt | ~1.15pt |
| `\ding{73}`（rEXP，对照组） | 10pt | 8.23pt | 8.23pt | 0.00pt |
| `\ding{73}`（rEXP，对照组） | 12pt | 9.87599pt | 9.87599pt | 0.00pt |

- **后果**：对照组 `\ding{73}` 因为是基于文本的静态加载，所以在两个时期测宽完全一致；而 `$\star$` 在类加载期少算了近 1pt 的宽度，导致实际排版时预留空间不足，侵入了星号前的列间距（colsep）。
- **修正方案**：必须将两个标记盒子的 `\hbox_gset:Nn` 测宽操作移至 `\AtBeginDocument`（或等价的 `begindocument` 钩子）中，在数学字体完全激活后进行延迟动态测量。

### 4. 仓库特定变体约定（e-type 优先）

- 在提示超长函数的接口设计中，新增的警告提示宜遵循仓库的规范约定，将原有的 `\msg_note:nnx` 升级为 **`\msg_note:nne`**（e-type 优先）。
- 内部继续保持 `\tl_to_str:n {#1}` 的防御性处理，防止用户在参数中传入特殊反斜杠字符引发无限宏展开崩溃。

## 影响到的稳定认知

- **类加载期测宽不等于运行时安全**：在涉及数学公式符号（如 `$\star$`）的宽度测量时，必须时刻警惕 `unicode-math` 等宏包的“延迟激活”机制。类加载期或导言区的静态测量往往会掉入默认字体的陷阱，**数学符号测宽必须在文档期（`\AtBeginDocument`）进行**。
- **作用域命名契约是刚性红线**：在 expl3 开发中，全局操作（`_gset`）绝不能施加在局域变量（`\l__`）上。重构时必须全局搜索，确保声明、赋值、使用三处的前缀契约完全收敛。
- **文档规约必须与代码强同步**：代码逻辑发生实质变更时（例如从 5 档变 3 档、静态宽度变动态测宽），对应的设计规约文档（如 `coding-conventions.md`）必须在同一个 PR 中闭环更新，防止过时文档误导后续开发者。
