# 456: xeCJK 长标点断点两侧禁则修复

## 任务

修复 xeCJK issue #456：长标点（`——`/`……`）与其他标点相邻时，`\@@_punct_kern:NN` 只要一侧是长标点就总允许折行，未检查断点另一侧是否违反标点禁则，导致三类硬性违规（`“——` 左标点悬尾、`——，`/`——。` 右标点落首、`——……` `NoBreakLongPunct` 落首）。

## 根因

v3.6.0（2018/01/23）把决策树设计为"一侧是长标点即可断"，这是对"长标点内部相邻"这一狭义场景的优化（避免长标点自身连用时无法断行），但把这个判定错误地推广成了"长标点与任意标点相邻都总可断"，丢失了断点另一侧的禁则信息。v3.10.0（2026/04/27）修了"右侧是长标点"分支下对 `NoBreakLongPunct` 的检查，但决策树仍是单侧检查结构，没有覆盖"左侧是长标点、右侧是 `NoBreakLongPunct`"的对称组合——这是一个典型的"补丁只覆盖复现路径，未审视决策树对称性"的遗漏模式。

## 关键坑位

### `\@@_punct_if_right:N` 与 `\@@_punct_if_long:N` 的参数形态不同，混用需要分别处理

`\@@_punct_kern_break:NN` 需要同时用这两类条件判断同一个 `#1`（来自 `\g_@@_last_punct_tl`，tl 类型）：

- `\@@_punct_if_right:N`（`prg_new_conditional` 生成，内部 `\xeCJKtoken_value_class:N` 查 `\XeTeXcharclass`）要求参数是**字符记号**（char token），不能直接吃 tl 变量。
- `\@@_punct_if_long:N`（special punct clist 机制生成，内部 `\if_cs_exist:w` 判断缓存 csname 是否存在，csname 由 `\@@_special_punct_tl:nN` 用 e-type 拼接）可以**直接吃 tl** 作为 `#`-参数。

第一次写 `\@@_punct_kern_break:NN` 时若把两个条件不加区分地都直接喂 `#1`，`\@@_punct_if_right:NTF #1` 会因为 `#1` 是 tl（非单字符 token）而在 `\xeCJKtoken_value_class:N` 处出错或得到错误的类别判断。正确写法是仅在调用 `\@@_punct_if_right:NTF` 前加 `\exp_after:wN` 把 `#1` 展开为字符记号，调用 `\@@_punct_if_long_p:N` 时保持原样传 tl。

**教训**：expl3 里两个名字都叫"判断某个标点属性"的条件函数，参数形态可能完全不同（char token vs tl），取决于它们各自底层依赖的原语（`\XeTeXcharclass` 需要字符层面的记号，`\if_cs_exist:w` 只需要能构造 csname 的任意 token 序列）。混用前必须逐一确认，不能假设"同一批标点判断函数"参数形态一致。

### `\@@_punct_kern_break:NN` 只做判断，不复制 kern 计算逻辑

延续 `\@@_punct_kern:NN` 原有"先选函数名，尾部统一喂参数"的结构，新函数只留下 `\@@_punct_breakable_kern:NN` 或 `\@@_punct_nobreak_kern:NN` 这个函数名，不在分支内部直接执行 kern。这避免了在两处重复实现已有的 kern 计算，也保持了与现有代码风格一致。

## `punct.tlg` 基线联动 diff 的验证方法：变化模式一致性

ctex 的 `punct.tlg` 是一个 180+ 测试的大文件，本次修复后跑出一批 diff。判断这批 diff 是否是"预期的修复效果"而非"引入了新 bug"，用的方法是**检查所有变化点的位置与模式是否一致**：

- 所有 diff 位置都发生在 `…` 后紧跟 `」` 的地方（即"省略号+右引号"这个具体标点对）。
- 节点变化模式完全一致：`\rule(0.0+0.0)x<width>` + `\glue <width> minus <shrink>` → `\penalty 10000` + `\glue 0.0 minus <shrink>`（`\rule` 的宽度分量被吸收进 penalty 前的 0-glue，原有的 shrink 分量保留不变）。

`\rule` 到 `\penalty 10000` 的变化直接对应"断点从可断（rule 只是零宽或负宽垫片，无禁则保护）变为禁则保护（penalty 10000 阻断该处断行）"——这正是本次修复的目标行为（`……」` 属于"长标点 + 全角左标点"组合，右侧 `」` 是 `FullLeft`，按新规则左侧长标点悬尾会被禁止，即断点应被禁）。

**方法论**：面对大文件基线联动 diff，不要求逐条人工确认排版效果，而是抽象出"变化点位置的共同特征"+"节点变化的统一模式"，只要两者都收敛到与本次修复目标一致的单一解释，就可以判定为预期变化，直接 `l3build save` 更新基线。如果 diff 位置分散、模式不一致，则说明可能引入了非预期的连带影响，需要逐条排查。

## showbox 禁则矩阵调试法

修复前后用 10 组标点对做系统性验证：`“——`、`——”`（同 `“——` 对称方向不适用，实际用左引号/括号系列）、`——，`、`，——`、`（——`、`——）`、`——……`、`……——`、`，……`、`“……`。对每一组构造 `\showbox` dump，逐字符对提取中间节点，按以下规则判定：

- 中间是 `\penalty 10000` 保护的 glue → **nobreak**（禁则生效）
- 中间是无 penalty 保护的 glue（或 `\rule` + 无保护 glue）→ **BREAK**（可断）

修复前的矩阵显示 8 处非法可断点（对应三类违规的具体标点对组合），修复后矩阵收敛为：所有非法组合变为 nobreak，2 处合法断点（`，——`、`……——`）仍保留可断。

**方法论价值**：相比"整篇排版后肉眼看断行位置"，矩阵法把"标点对 × 断点状态"做成一个显式的二维表格，能够：
- 系统性覆盖所有需要验证的标点对组合，避免遗漏对称情形（如只测了 `——，` 忘测 `，——`）；
- 精确定位断行语义在节点级别的证据（penalty 值），而不是依赖视觉判断断行位置是否"看起来正确"；
- 修复前后各生成一次矩阵，直接对比收敛过程，天然构成回归测试用例设计的依据（`longpunct-kinsoku01.lvt` 的用例组织直接对应矩阵中的违规/合法分组）。

这一方法与既有反思 [[284-fullwidth-tilde-longpunct]]、[[671-cjkpunct-rglue-nobreak]] 中"`\showbox` 是标点间距问题的核心分析方法"一脉相承，本次的增量贡献是把单点 dump 提升为系统性矩阵扫描，适合"断点两侧禁则"这类需要覆盖多个标点类型组合的判断逻辑。

## 教训

- 决策树类修复要检查对称性：一个分支被发现有遗漏并打了补丁后，应该反向审视决策树的镜像分支是否有同类遗漏（本次 v3.10.0 只补了"右侧是长标点"分支，未联动检查"左侧是长标点"的对称组合）。
- expl3 中同属"标点属性判断"命名族的条件函数，参数形态（char token vs tl）可能不同，取决于各自依赖的底层原语；混用前逐一确认，需要展开的地方显式 `\exp_after:wN`，不需要的地方不要画蛇添足。
- 面对大文件基线联动 diff，用"变化位置的共同特征 + 节点变化的统一模式"两条收敛证据判断是否为预期变化，比逐条人工核对排版效果更快、更不容易遗漏连带副作用。
- 断点两侧禁则类问题适合用"标点对矩阵 + `\showbox` 节点判定"做系统性调试，覆盖面天然指导回归测试用例设计。

## Follow-up

- 未来若再有标点断行/禁则类 issue，可复用本次的"标点对矩阵"方法预先枚举所有需要验证的相邻标点组合，而不是逐个用户报告的具体场景补丁。
- `\@@_punct_kern:NN` 决策树中"两类条件函数参数形态不同"这一约束具有一定普遍性，若未来标点系统还有新的条件函数加入，应在实现前先确认其参数形态（char token / tl / cs 等），必要时在架构文档中补一条通用提示。
