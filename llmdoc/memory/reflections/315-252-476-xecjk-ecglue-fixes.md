## 任务

补齐 xeCJK 中另一类长期存在的边界恢复问题：当 `\texttt`、`\textbf`、`\textit`、`\zihao` 等命令在组内切换字体或字号后，紧随其后的前侧 `CJKecglue` 会使用组内字体的空格度量，而不是外层 CJK 字体的度量，导致 CJK 与 Boundary 之间的间距在视觉上忽大忽小。该问题对应 issue #252 与 #476，并与 #315 一起在 PR #791 中统一修复。

## 根因

### `\CJKecglue` 默认是 `~`，因此它本质上是 font-dependent glue

这次修复最关键的认知转变是：`\CJKecglue` 不是一个固定尺寸常量，而是默认展开为 `~`（非断空格）。而 `~` 的 glue 值来自当前字体的 `\fontdimen`，所以同样一个 `\CJKecglue`，在 Roman、Typewriter、Bold 或不同字号下的宽度、stretch、shrink 都可能不同。

因此，问题的真正根因不是“xeCJK 忘了恢复 ecglue”，而是“xeCJK 在前侧边界恢复时重新展开 ecglue 时，所处的字体上下文已经不对了”。如果 CJK→Boundary 过渡中间夹着一个局部分组，而这个分组切换了字体，那么恢复时直接再执行 `\CJKecglue`，拿到的就会是分组内部字体的空格度量。

### 两个看似不同的 bug，实际上共享同一段恢复路径

- #315 关注的是：边界标记前插入 whatsit 后，`\lastkern` 检测链断掉，导致 glue 根本没有被恢复。
- #252/#476 关注的是：glue 虽然恢复了，但恢复时重新测量，拿到了错误字体的度量。

二者表面症状不同，但都发生在 xeCJK interchar 状态机的“边界恢复路径”上：前者是“能不能恢复”，后者是“恢复时拿什么值恢复”。这也是为什么它们最终适合放进同一个 PR 里一起修，而不是拆成完全无关的两个补丁。

## 关键实验方法

### 1. 先做 box dump 对照，确认问题在 glue 而不在 glyph

排查这类间距问题时，第一步不是盯着宏展开，而是先把有问题和无问题的输入分别装进盒子，比较节点列表或总宽度。box dump 能先回答一个关键问题：变化的是字符本身，还是字符之间插入的 glue / kern。

对于这次问题，box dump 对照能很快排除“某个字体切换命令把字形换宽了”的干扰，把注意力集中到边界恢复出来的那段 ecglue 上。

### 2. 再用 `\meaning` 看 `\CJKecglue` 的真实定义

如果只凭直觉，很容易把 `\CJKecglue` 当成一个固定 skip。真正把 `\meaning\CJKecglue` 打出来后，才能确认它默认其实落到 `~`，也就自然推导出 glue 度量会跟当前字体走。

这一步非常重要，因为它把调试方向从“xeCJK 的 glue 参数是否被覆盖”转成了“恢复时的字体上下文是否正确”。

### 3. 用 `\fontdimen` 验证不同字体下空格度量确实不同

当怀疑 `~` 是 font-dependent 之后，最稳的验证方式不是靠肉眼看 PDF，而是直接比较不同字体/字号下的 `\fontdimen`。只要能确认 `\texttt`、`\textbf`、缩小字号等上下文里的空格宽度、stretch、shrink 与外层 CJK 字体不一致，就能证明“恢复时重新展开 ecglue”在设计上必然不稳定。

这类验证比仅做最终宽度比对更强，因为它直接证明了底层度量来源变了。

## 修复决策

### 为什么选“CJK→Boundary 时缓存”，而不是初始化时缓存或每字符缓存

最终方案是在 `\@@_boundary_group_end:n` 里，也就是 CJK→Boundary 过渡时，把当下正确字体上下文中的 `\CJKecglue` 先缓存到 `\l_@@_ecglue_skip`，后续所有前侧边界恢复路径统一使用这个缓存值。

这个时机优于另外两种看似可行的方案：

- 不是初始化时缓存：因为用户可以在文档中途切换 CJK 主字体、字体族或字号，初始化阶段得到的值不再可靠。
- 不是每个字符都缓存：虽然也能工作，但频率过高，且会把状态机复杂度推得更高。

选择 CJK→Boundary 时缓存的原因是：

1. 这正好是“即将离开 CJK 字体上下文”的最后稳定时机；
2. 此时测到的 ecglue 值仍属于外层正确的 CJK 字体；
3. 只在边界处缓存，频率适中，不必给每个字符都附带额外成本；
4. 后续恢复路径无论是否经过字体切换分组，都可以使用同一份已测得的 skip 值。

### 修复后的心智模型

修复后，xeCJK 的前侧边界恢复不应再理解为“需要时再执行一次 `\CJKecglue`”，而应理解为：

1. 在 CJK→Boundary 离开 CJK 区域时，先测量并缓存当前应使用的 ecglue；
2. 后续如果需要在边界前侧恢复 ecglue，就消费缓存值；
3. `\lastkern` / whatsit 回退链负责判断“该不该恢复”；ecglue 缓存负责保证“恢复时用哪个值”。

也就是说，#315 与 #252/#476 合起来后，这条状态机才完整覆盖了“判定”和“取值”两个维度。

## 测试设计经验

### “前侧 ecglue 只测量”不能直接比较总宽度

这次一个很重要的测试经验是：当前侧 ecglue 的目标是测量“边界前恢复出来的那一段 glue”时，不能只比较整个盒子的总宽度。因为一旦测试里同时切换了字体，glyph 本身的宽度也会变化，直接比较总宽度会把字形变化和 ecglue 变化混在一起，导致结论不干净。

更可靠的做法是把测试设计成“只隔离前侧 ecglue 的贡献”，必要时从总宽度里减去对应 glyph 的宽度，或者让对照组与实验组共享完全相同的 glyph，只让边界恢复链不同。核心经验是：

- 总宽度比较适合验证“用户可见输出是否整体一致”；
- 但若目标是定位某一段前侧 ecglue，必须把 glyph 宽度的变量剥离掉。

这也是 `xeCJK/testfiles/fontswitch01.lvt` 能稳定覆盖 `texttt`、`textbf`、`textit`、`small` 这几类场景的关键。

## CI 教训

### 基线连环失败说明本地必须先按 CI 矩阵完整跑一轮

这次修复最终影响了 ctex 22 个基线，原因不是 patch 错了，而是 ecglue 现在统一回到 CJK 字体的空格度量后，stretch/shrink 的细小数值与旧基线不同。它们在单个测试里看起来只是微量变化，但一进 CI 矩阵，就会连锁放大成多处 baseline mismatch。

教训是：只在当前子包里跑最小回归还不够。只要修改的是 xeCJK 的边界恢复链，就要预期 ctex 这种依赖 xeCJK 的上层包也会受影响，而且不同平台/字体环境下更容易把细小差异显出来。

因此，处理这类底层间距修复时，推送前应尽量在本地按 CI 矩阵完整跑一轮，至少把直接受影响的主包和卫星包都检查到位，避免进入“修一处、炸一片、再逐个补基线”的被动节奏。

## 教训

- 遇到 xeCJK 边界空白问题时，不仅要问“有没有恢复 glue”，还要问“恢复出来的 glue 是在哪个字体上下文里测量的”。
- 对默认定义依赖字体度量的宏，`\meaning` 与 `\fontdimen` 验证往往比盯着宏名本身更能揭示真实根因。
- 只要修改的是 interchar 边界恢复链，就应主动检查同一区域是否还承载别的历史 bug；这次 #315 与 #252/#476 共用恢复路径，就是一个典型例子。
- 涉及底层 glue 度量的修复即使视觉变化很小，也可能引发大面积 `.tlg` 更新；本地验证范围应按 CI 依赖链来设，而不是只按当前改动目录来设。
