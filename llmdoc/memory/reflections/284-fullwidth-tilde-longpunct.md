## 任务

为 Issue #284 / PR #793 记录修复反思：处理 xeCJK 中全角波浪号 U+FF5E（`～`）前后多余空白的问题，并总结为什么最终应把 U+FF5E、U+301C、U+30A0 从 `MiddlePunct` 改为 `LongPunct`。

## 背景修正

### v3.6.0 已基本修复“可见间距”问题

分析时首先要修正一个容易过时的认知：`～` 的问题并不是一直都表现为明显的可见空白。

自 v3.6.0 起，xeCJK 已将 U+FF5E 从普通 CJK 字符改为 `FullRight + MiddlePunct`。这一步已经基本修复了最直观的视觉问题：字符不再按普通汉字路径参与两侧间距处理，因此多数场景下用户看到的“多余空格”现象已经大幅缓解。

剩余问题更细：`MiddlePunct` 这一路径仍会触发标点压缩相关的 glue/rule 节点构造。虽然这些节点的总宽度在很多情况下会互相抵消，不一定形成新的可见空白，但它们仍然会给本不应进入标点压缩流程的连接号引入额外排版节点和 shrink 分量。

因此，这次修复的重点不是“第一次解决明显空格”，而是把字符分类从“基本可用”进一步修正到“语义与节点模型都更准确”。

## 关键发现

### `MiddlePunct` 的问题不在总宽度，而在不必要的节点模型

`MiddlePunct` 类字符会走 xeCJK 的标点压缩逻辑，在字符两侧插入标点相关节点，典型表现为 punct glue 与 negative-width rule 的组合。

这组节点的一个迷惑点在于：从总宽度看，它们经常彼此抵消，所以仅凭最终盒子宽度不一定能立刻看出异常。但从排版结构看，这依然意味着：

- 连接号类字符被当作“需要压缩的中点号/间隔号式标点”处理；
- 水平列表中出现了额外的 glue/rule 节点；
- glue 还可能携带 shrink 分量，影响断行与伸缩行为分析；
- 即使肉眼看不到多出的一整块空白，节点层面仍不够干净。

对 `～`、`〜`、`゠` 这类更接近连接号或波浪连接号语义的字符来说，这种处理粒度过粗。

### `LongPunct` 与 `MiddlePunct` 不是“只差一个名字”

分析中一个重要结论是：把字符改到 `LongPunct` 后，并不是简单地换了一个分类标签，而是切换到了另一套宽度计算与节点生成路径。

`LongPunct` 字符会以原始宽度排版，不再经过 `MiddlePunct` 那套压缩流程；同时依然允许前方断行，并可继续保留 `FullRight` 提供的行首禁则语义。这正好匹配本次修复目标：

- 不需要中点类压缩；
- 仍保留右侧全角标点的边界约束；
- 维持更符合连接号语义的排版行为。

### `\@@_CJK_and_FullRight_glue:N` 中 `punct_if_long` 与 `punct_if_middle` 是独立判断

这次分析里一个非常关键、也很容易先入为主误判的点是：

在 `\@@_CJK_and_FullRight_glue:N` 中，`punct_if_long` 与 `punct_if_middle` 不是一个互斥的 `if / else` 分支，而是两个独立判断。

这意味着在理解某个字符的 glue/rule 生成行为时，不能简单假设“它是 long 就不会再碰 middle 路径”，或者反过来只凭一个分类名去脑补完整控制流。必须直接对照实际条件判断与最终节点输出。

也正因为这两个判断不是概念上的二选一，分类调整后对节点模型的影响需要通过实际盒子输出来验证，而不能只靠阅读名称推断。

### `LongPunct` 的零宽计算会得到更干净的输出

对 `LongPunct` 字符，相关 glue/rule 的宽度计算会落到零，因此最终得到的是更干净的排版结果。分析时可对照已有 `LongPunct` 字符，如 U+2014 `—`：

- 它不会像 `MiddlePunct` 那样留下不必要的标点压缩节点痕迹；
- 盒子结构更接近“字符按原始宽度直接排版”；
- 在调试日志里更容易确认没有多余 shrink / negative rule 参与。

这也是本次把 U+FF5E、U+301C、U+30A0 改为 `LongPunct` 的核心依据之一：不仅视觉上正确，而且内部节点模型也更简洁。

## 调试方法

### `\showbox` dump 是分析标点间距问题的核心技术

这类问题如果只看源码中的分类名、宏名或最终 PDF 视觉效果，很容易误判。

真正可靠的方法是直接比较 `\showbox` dump，观察字符附近进入水平列表的实际节点：

- glyph
- glue
- kern
- penalty
- rule
- 以及它们各自的 width / stretch / shrink

只有看到完整节点序列，才能判断：

- 是否真的插入了额外标点压缩 glue；
- negative-width rule 是否参与了总宽度抵消；
- 问题是“可见空白”还是“节点结构脏但宽度表面正常”；
- 分类调整后是否真正切换到了更干净的排版路径。

对 xeCJK 的标点间距问题，`\showbox` 对比不只是辅助技巧，而应视为核心分析方法。

## 可复用教训

- 当某个字符的视觉问题在旧版本与新版本表现不同，先确认它是否已在历史版本中被部分修复，不要把“残留问题”误写成“原始问题未修”。
- xeCJK 标点分类问题不能只看最终宽度；即使总宽度抵消，额外 glue/rule 节点和 shrink 仍可能说明分类不当。
- 阅读 `punct_if_long`、`punct_if_middle` 这类条件时，不要先验假设它们互斥，必须结合真实控制流与 `\showbox` 输出验证。
- 对连接号、波浪号这类边界字符，分类选择的标准不只是视觉效果，还包括是否引入了不必要的标点压缩节点模型。
