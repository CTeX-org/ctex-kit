# [Task Reflection]

## Task

在 #992 建立的 capture/register 框架之上增加实验性用户接口
`experiment/boundary-register`。用户直接提供一个控制序列，并从 `box`、
`wrapped-box`、`stream`、`transparent`、`post-transparent` 五种策略中选择
与目标命令节点结构相符的一种；适用时还可用 `auto`、`default` 或
`first-default` 声明首尾类别的取得方式。

接口只允许在导言区新增注册，不提供反注册，也不允许覆盖 xeCJK 已有的通用注册
或专用适配器。声明先全局收集，到导言区末尾统一应用；普通
`\AtBeginDocument` 中才定义的命令仍可注册，正文开始后才定义的命令则报告错误。
随包示例用直接输入作为比较基准，展示四种源码空格、左右边界和
`xCJKecglue=false/true`，并明确展示通用接口不能自动处理参数内部公式边界这一
负例。

## Expected vs Actual

- 预期：公共接口的主要工作是增加一组 l3keys，把 `command`、`strategy` 和
  `mode` 转交给既有注册函数。
- 实际：键值解析只是表面工作。真正困难的是向用户说明每种策略对应什么可观察的
  节点结构，以及声明何时保存、何时安装 hook、何时判断命令是否存在和是否冲突。
  策略选错或时机处理不当，都可能破坏目标命令本身，而不只是得到错误间距。
- 预期：既有的通用注册属性表可以回答“这个命令是否已由 xeCJK 处理”。
- 实际：`\verb`、`\Url@z`、codedoc、ulem 和 listings 等入口使用直接重定义或
  专用扫描适配，并不都经过通用注册函数。首轮独立审计据此发现，用户仍能给
  `\verb` 叠加通用 `stream` hook，正文会以 `Improper alphabetic constant`
  终止。
- 预期：用户声明时检查一次冲突即可。
- 实际：声明可能早于 `url`、hyperref 等宏包加载。保存时没有冲突，不代表导言区
  末尾仍没有冲突；必须在保存和实际应用两个阶段都检查，才能覆盖晚加载宏包和
  延迟建立的专用适配器。
- 预期：`auto` 既然观察实际输出，就能处理所有首尾内容。
- 实际：普通 interchar 观察看不到参数内部的公式语法和公式节点确认过程。
  #1002 的能力来自各命令参数适配器在可见正文排完之前进行的专用确认，并不是
  五种通用策略自带的能力。公共接口不能把公式边界宣传成 `auto` 的一般能力。

## What Went Wrong

1. **最初把“通用注册表”误当成了“全部已处理命令”。**
   `\g_@@_boundary_registered_prop` 只记录通过五种通用注册函数安装的 hook，不能
   表示直接重定义扫描器或内部排版入口的专用适配器。首轮隔离审计用 `\verb`
   证明这不是抽象的完整性问题：重复处理会直接破坏特殊参数读取。

2. **冲突检查如果只有一个时点，就会漏掉加载顺序。**
   用户可能先声明 `\url`，再加载 `url` 宏包。只有声明时检查会允许记录进入待应用
   表；只有应用时检查又不能尽早拒绝已经确定的内建冲突。最终实现让保存阶段和
   导言区末尾应用阶段查询同一个“通用注册＋专用保留”判断。

3. **容易把 `auto` 理解成对参数语义的自动分析。**
   它只使用 capture 运行时能够观察到的类别。命令以公式开始或结束时，源码扫描
   只能产生候选，实际末节点还要在目标命令的参数处理位置确认；未知宏甚至可能
   消费 `$`、`\)` 或整个公式分组。#1002 为内建命令增加的这层适配不能由一个
   普通命令 hook 自动取得。

4. **诊断测试一度可能只证明内部状态，没有固定用户实际看到的消息。**
   begin-document 阶段的输出若发生在 `\START` 之前，就不会进入 `.tlg`。最终测试
   在日志捕获开始后再次执行存在性检查，使“目标命令仍未定义”这一公共诊断真正
   成为回归基线；初始化噪声则用 `\OMIT`／`\TIMO` 隔开。

5. **总宽度相等仍可能掩盖边界错误。**
   左侧多出的间距可以与右侧缺少的间距互相抵消。因此用户接口的行为测试除
   `00/10/01/11` 外，还分别构造 left-only 和 right-only 比较，不能只依赖整段
   总宽度。

6. **示例如果只展示成功项，会让接口能力看起来比实际更强。**
   公式边界恰好是通用注册的重要限制。可视 MWE 保留公式行的不等宽结果，并说明
   它需要 #1002 那类参数适配；负例在这里是接口说明的一部分，不是应当隐藏的失败。

## Root Cause

根因是把三个不同层次混在了一起：五种策略描述命令在节点列表中留下什么，用户
注册的生命周期决定何时能够安全安装 hook，而“xeCJK 已经处理某命令”还包括不走
通用 hook 的专用扫描和排版适配。单一注册属性表只能回答第一类实现路径，无法同时
表达后两类事实。

公式限制也是同一个边界的另一面。capture/register 可以统一恢复算法，却不能从
通用命令 hook 推断任意参数如何被消费。`auto` 的“自动”只表示使用框架实际观察到
的首尾类别，不表示分析未知命令的参数语义。

## What Worked

- 公共接口只保存策略选择，最终仍调用既有五种注册函数，没有复制边界恢复算法。
- 新增独立的专用适配器保留表，并让统一判断同时查询通用注册表和保留表。核心
  命令、颜色、hyperref、URL、verb、codedoc、ulem 和 listings 的专用入口均登记
  保留名。
- 保存和应用两个阶段使用同一冲突判断。独立晚加载实验先声明 `\url`、后加载
  `url`，导言区末尾能够拒绝冲突，证明第二次检查覆盖了加载顺序。
- `command` 直接接收一个控制序列，`@`、`_` 和 `:` 的类别码由用户分别用
  `\makeatletter` 或 `\ExplSyntaxOn` 管理。接口不再引入命令名字符串解析这一层
  额外规则。
- 行为测试覆盖五种策略、三种允许的 mode、分组内声明、普通
  `\AtBeginDocument` 定义、`00/10/01/11`、左右单边界和
  `xCJKecglue=false/true`，共完成 288 项比较且没有失败。
- 诊断测试覆盖非控制序列、缺项、非法选择和组合、重复声明、通用内建冲突、专用
  适配器冲突、未定义目标和正文期声明，并在拒绝 `\verb` 的重复注册后实际调用
  `\verb`，确认专用扫描器仍可使用。
- 可视 MWE 用明显不同的 `CJKecglue` 和 `CJKglue` 帮助观察结果，同时分别展示
  中西和西中两个方向，使读者能看见左右边界而不是只看总宽度。
- 本地独立审计以固定提交和不继承主对话的快照进行。首轮唯一的重要问题没有被
  后续“全绿”总评掩盖：修复后先做隔离增量审查，再把独立证据映射回 F-001，最后
  从原始基线做全范围盲审；终端收据确认最终为 0／0／0 且证据链完整。

## Missing Docs or Signals

- 既有架构文档说明了五种内部注册策略，但此前没有稳定记录用户接口的声明窗口、
  双阶段冲突检查、专用适配器保留表和正文开始前存在性检查。这些内容应进入新的
  #1010 决策文档和 xeCJK 架构说明。
- 构建与测试参考尚未记录 `boundary-register-api01/02` 的职责：前者固定行为矩阵，
  后者固定公共诊断与生命周期。尤其应说明 begin-document 诊断必须在 `\START`
  后实际进入 `.tlg`。
- #992 决策中的“注册表拒绝重复注册”主要描述内部通用注册，容易让读者误以为该表
  已覆盖全部专用适配器。稳定文档应明确区分通用注册表和专用保留表。
- 用户手册已经说明公式负例和专用扫描器限制；后续修改接口时，应继续保留这些限制，
  不能只保留成功示例。

## Promotion Candidates

- **公共注册接口的核心是策略语义和生命周期，不是键值语法。** 文档必须先让用户
  判断目标命令的节点结构，再说明声明、应用和存在性检查的时点。
- **“已处理命令”不能只由通用注册表表示。** 直接重定义或专用扫描适配器需要独立
  保留表；用户冲突检查应查询两者的并集。
- **延迟加载场景需要双阶段冲突检查。** 保存阶段尽早拒绝已知冲突，应用阶段再次
  检查导言区内后来加载的宏包。
- **`auto` 只观察框架可见的类别，不分析参数语义。** 参数内部公式等边界仍需专用
  适配器，不能因恢复算法只有一套就宣称适配器也能完全通用。
- **边界回归必须分别检查左右两侧。** `00/10/01/11` 与总宽度比较不足以排除两侧
  误差抵消。
- **公共错误消息必须落入回归日志。** 若诊断发生在 `\START` 前，应在捕获区间内
  重放相同检查；用 `\OMIT`／`\TIMO` 隔离初始化输出，而不是只检查内部变量。
- **能力边界应由可复现负例说明。** MWE 可以保留明确解释的失败行，帮助用户识别
  何时应停止尝试通用策略并改写目标命令或等待专用适配。

## Follow-up

- recorder 应把用户接口的稳定契约写入 xeCJK 架构、#992 决策、构建与测试参考，
  并新增 #1010 决策文档；同时更新 `llmdoc/index.md`。
- PR 说明应以既定实施决定为背景，展示接口、五种策略、可视 MWE 和公式限制，不能
  写成再次征求是否需要该接口。
- 后续若登记新的内建专用适配器，必须同时把其公开入口和实际内部排版／扫描入口
  加入保留表，并补一个能证明专用行为未被通用 hook 破坏的测试。
- 后续若要扩展公式能力，应先设计相应参数适配器和直接公式 oracle；不能扩大
  `auto` 的文字承诺来代替实现。
