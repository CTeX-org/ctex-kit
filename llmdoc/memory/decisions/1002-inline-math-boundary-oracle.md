# 决策：#1002 以直接公式作为行内公式边界的比较基准

## 背景

#992 用“命令包装与相同可见内容的直接输入等价”统一普通中文、西文和数字的命令边界，但最初没有系统覆盖行内公式。直接输入 `$x$`、`\(x\)`、`\ensuremath{x}`，以及 `\textbf{$x$}`、`\mbox{$x$}`、`\fbox{$x$}` 等命令包装，会经过不同的数学模式和盒子路径，不能用普通西文字母的结果代替。

固定提交 `4628cb44`（xeCJK v3.10.3）与 `10500b33`（PR #1001 合并后的 `master`）给出相同的直接公式结果，说明当前差异不是 PR #999 或 #1001 引入的回归。

## 比较基准

对 `\foo{$x$}` 只去掉外层命令，oracle 必须保留为 `$x$`：

```tex
中\foo{$x$}文
中$x$文
```

两行应具有相同的左右边界间距。不能把 oracle 换成字母 `x`；那会丢掉公式自身的数学模式和源码空格语义，并可能把同一项测试从失败变成通过。

测试名称使用“中文—公式—中文”和“西文—公式—西文”，不把公式归入普通数字场景。行间公式通常不与同一行的左右正文形成这种边界，不属于本决策的默认范围；如有具体场景，应单独说明输入形式和 oracle。

## `xCJKecglue` 语义

保留现有选项含义：

- `xCJKecglue=false`：公式旁的源码空格保留为普通词间空格；
- `xCJKecglue=true`：公式旁的源码空格改用 `CJKecglue`。

候选和 oracle 必须使用相同的选项值。外层命令不能改变用户选择的空格处理方式。手册应明确说明，`xCJKecglue` 不只影响普通西文，也控制行内公式旁的源码空格。

## 支持顺序

`$x$` 是最常用的行内公式形式，优先完整支持以下路径：

- 直接输入 `$x$`；
- 显式分组 `{$x$}`；
- 字体和颜色命令中的 `$x$`；
- `\mbox`、`\fbox`、`\colorbox` 等盒子中的 `$x$`。

`\(x\)` 和 `\ensuremath{x}` 的目标行为与 `$x$` 相同。实现已经把三种写法接到同一个 `math` 边界类别；以后若增加新的公式包装，也必须保留相同 oracle，不能把公式降为普通西文字母。

## 实现约束

capture/register 框架为行内公式增加独立的 `math` 类别，但只保留一套边界恢复算法。命令适配器负责识别可见正文两端的公式语法：正文以 `$`、`\(`、`\ensuremath` 或只含这些内容的外层分组开始时，在 math-on 前报告首类别；正文尾部识别出的 `$`、`\)`、`\ensuremath{...}` 或相应外层分组只产生公式候选，不能直接把当前最内层 capture 的末类别标为 `math`。

所有尾部公式语法都需要两阶段判断。第一阶段只检查源码，产生“可能以公式结束”的候选；这一步不能发布末类别，因为未知宏既可能在可选参数、普通参数或分隔参数之后消费尾部花括号组，也可能直接把 `$` 或 `\)` 当作分隔参数的终止符。第二阶段的确认代码位于已注册命令的可见正文参数末尾：正文实际排完、外层包装尚未关闭时，只有当前列表末尾仍是真实 math 节点或 xeCJK 的 `math` marker，才把候选发布为 `math`。例如 `中{$x$}`、`中$x$` 会通过确认；若未知宏消费 `[q]{$x$}`、末尾 `$` 或末尾 `\)` 而只排出 CJK“文”，列表末尾是 CJK marker，候选便被丢弃。这样无需理解未知宏的参数结构，也不会展开它们。

参数正文以“公式＋源码空格”结束时，空格 glue 会遮住公式节点。通用适配器只在源码扫描已经确认存在尾随空格、且末节点确为普通词间 glue 时，暂时取下这枚 glue 检查下方的公式节点，再原样放回；不能越过任意显式 glue。ulem 会先把源码空格改造成 leaders，因此它的正文扫描器先排去掉尾随空格的正文，在公式节点仍可见时完成同一确认，再把一枚空格交还 ulem 排版。

确认成功后，框架按参数内空格是否继续参与外层断行，发布两种不同的末尾 marker：

- 普通 `stream` 使用 `math-space`。参数内的实际空格仍位于外层列表，marker 用两对零净宽 kern 保存它的伸长量和收缩量（stretch/shrink）。右侧为 CJK 且 `xCJKecglue=true` 时，补偿的自然宽度为 `CJKecglue` 自然宽度减入口普通词间距自然宽度；弹性部分完整取 `CJKecglue` 的伸缩量，再扣除实际 stream 空格已有的伸缩量。内部字体改变所造成的空格自然宽度差属于命令自身的固有宽度，不在边界补偿中消除。
- `box`、`wrapped-box` 和 `stream-ulem` 使用 `math-space-frozen`。参数内空格已经冻结在盒子或装饰内容中，不参与外层断行；外层补偿因此完整保留 `CJKecglue` 的伸长量和收缩量，自然宽度为 `max(CJKecglue 自然宽度 - 入口普通词间距自然宽度, 0pt)`。下限为零是必要约束：若 `CJKecglue` 更窄，负 glue 会把后续 CJK 文字拉进框线或装饰范围。

两种状态都保留参数内的原空格。命令外再有一枚源码空格时按连续源码空格处理；不符合词间空格形状的显式 glue 原样保留。

`math-space` 只在真实参数空格与 marker 仍物理相邻时有效。capture 入口保存 marker 前那枚真实空格的 skip；transparent 结束时，只有列表末尾仍是同值 glue，才重放 marker。若颜色 special、零尺寸 hbox 等不可见节点已经插到二者之间，含同一节点的直接公式 oracle 也不会跨节点恢复间距，所以 marker 应当过期，不能把补偿 glue 单独重放到节点之后。post-transparent 遇到 `math-space` 时同样保留原节点顺序并清除恢复状态，而不是沿用普通 marker 的后移逻辑。

末类别只写当前最内层 capture，而不是直接广播到全部外层。内层命令结束后会把 `math` marker 写回它实际所在的列表，外层 capture 再从真实 marker 逐层取得结果。因此，原语 `\setbox` 中没有输出到当前列表的公式不会污染外层命令；`\sbox` 仍使用既有 suspend/resume。实现不设置全局 `\everymath`，也不展开任意用户宏。

## 测试契约

稳定回归应覆盖：

- `xCJKecglue=false` 和 `xCJKecglue=true`；
- 默认间距和 5pt/1pt 可区分间距；
- `00`、`10`、`01`、`11` 四种源码空格；
- 中文—公式—中文和西文—公式—西文；
- 直接公式、等价公式写法和命令中的公式。

左右边界需要分别检查，不能只比较总宽度；一侧多出的间距可能与另一侧缺少的间距互相抵消。命令包装测试仍应扣除命令本身造成的固有宽度差。

## 当前状态

当前分支按上述设计完成实现和回归：

- `command-boundary01` 的四个旧跳过已改为实际断言，总数为 1668，失败数为 0；
- `command-boundary-math01` 在默认/可区分间距和 `xCJKecglue=false/true` 四种配置下执行 5504 次比较，失败数为 0；
- 测试覆盖 `$x$`、`\(x\)`、`\ensuremath{x}`，以及 box、wrapped-box、stream、stream-ulem、独立符号、外层分组、CJK 前缀加直接或分组公式后缀、嵌套命令和离线 `\setbox`；新增的尾随空格矩阵覆盖盒子、字体、颜色、链接和 ulem，并检查后接 CJK、Default、另一个注册命令和显式 glue；反例还覆盖未知宏消费公式后只留下 CJK 与尾随空格的 box/ulem 路径；
- `command-boundary-math05` 用带伸缩量的 `CJKecglue` 固定普通 stream 与冻结空格的节点契约，覆盖直接公式、box、wrapped-box、字体不同的 stream、嵌套 stream、嵌套 `\mbox` 和 ulem；另用比普通词间距更窄且不带伸缩量的 glue，确认冻结路径不会生成把后续文字拉进盒子或装饰范围的负间距；段落断言把 direct、box、wrapped-box、stream 和 stream-ulem 的自然宽度各缩短 1pt 后，五条路径的 badness 均为 12，证明 2pt 外层收缩量都可供段落装箱使用；颜色 special 和零尺寸 hbox 用例还确认 `math-space` 被隔开后应当过期，候选相对含同一节点的直接公式 oracle 宽度差为 0，段宽 10pt、容差 100 时段落高度差也为 0；
- `loading01` 将 `math-space`／`math-space-frozen` marker，以及计算补偿所需的 skip 和尺寸（dim）寄存器纳入分配基线；当前 xeCJK 标准回归测试为 108／108 通过；
- `gh-assets:issues/1002/` 的 `false-default`、`false-custom`、`true-default`、`true-custom` 四套矩阵均为 272／272；节点测试、宏包加载顺序、移动参数、对齐扫描器和只加载标准 `color` 的路径也通过。

这些结果说明当前代码树中的已登记命令已与直接公式 oracle 一致；它不表示可以推断任意用户宏的输出。#992 的公开活表仍只记录已合并实现，必须在 PR 合并后从合并提交重新验证，再更新对应状态和图片。
