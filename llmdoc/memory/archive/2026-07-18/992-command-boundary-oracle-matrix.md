# 反思：#992 命令边界的 oracle 与覆盖矩阵

## 单点通过不能代表整类修复

#491 的状态表按命令各选一个历史场景，适合展示定点补丁的演进，却把“这个矩阵单元通过”视觉上放大成“这个命令已经修复”。同一命令换成 CJK 输出、改变一侧源码空格或进入 restricted hbox 后，可能走完全不同的 interchar 与节点恢复路径。

长期状态必须记录“实际输出首尾类别 × 相邻字符类别 × `00/10/01/11`”的精确单元。命令名只负责定位实现入口，不是完成状态的粒度。

## 直接输入才是语义 oracle

命令候选应与去掉命令包装后的相同可见内容比较。这样西文/数字、CJK、混合和无可见输出的边界语义来自 xeCJK 已接受的直接输入，而不是测试作者对某个命令的预设。左右边界分别由实际首尾字符决定，不能从参数整体贴一个类别标签。

#991 进一步证明输出类别不能硬编码：`\@setref` 末尾的 `\null` 会遮蔽 marker，但引用结果可能是数字，也可能是 CJK。按数字 MWE 设计的 Default drain 即使修好一个单元，也可能破坏纯 CJK 无源码空格的最低保证。

## 默认宽度会制造假通过

默认词间空格和 `CJKecglue` 的自然宽度可能相同，默认 `CJKglue` 的自然宽度又可能为零。只看截图或总盒宽时，错误节点与正确节点可能恰好等宽。调查需要同时使用可区分的 5pt ecglue、1pt CJKglue、扣除命令固有宽度后的边界测量，以及节点级证据。

## 结果

#992 取代 #491，保留旧资料为历史证据，并把完整矩阵设为理想目标、两类场景设为最低保证。`gh-assets:issues/992/` 按 core、links、verb 和引用视觉矩阵重组素材；后续实现应把已支持的精确单元转成 `xeCJK/testfiles/` 回归，不再用每个命令一个成功示例宣称整类完成。
