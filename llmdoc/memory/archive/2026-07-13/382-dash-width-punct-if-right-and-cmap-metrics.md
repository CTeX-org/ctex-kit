## 任务

为 Issue #382 / PR #944（commit `8f67da87`，分支 `issue-382-dash-width`）记录修复过程中的关键教训：破折号宽度不符合 CLReq 两个汉字宽的要求，修复涉及压缩公式重构（三路取大 kern + margin 全份补偿）与新增 `PoZheHao` 字符类支持 OpenType 合字。

## 关键发现 1：`\@@_punct_if_right:N` 遗漏新字符类是"复用现有二元判断函数添加新分类"这类修改的典型陷阱

新增 `PoZheHao` 字符类后，最容易被忽略的一环不是"这个类要不要注入内容"（这是设计的核心目标，天然会被反复检查），而是**所有原本对 `FullRight` 做等值判断的下游函数**，是否也需要认识这个新类。

`\@@_punct_if_right:N` 原本只判断字符类是否等于 `FullRight`。当破折号被划入 `PoZheHao` 类后，这个判断对 U+2014 会返回假——即便在语义上 U+2014 依然应该被当作全角右标点处理。表现出来的失败场景是"标点 + 破折号"相邻（如"爱。——测"），下游代码按"不是全角右标点"的假设去查缓存表的键（形如 `dim/glue/left/—/tl` 一类由字符对拼接而成的属性表键），但因为分类判断错误，实际查的键并不存在对应的写入路径，最终报 `Missing number`（TeX 试图对一个空/未定义的宏值做数值运算时的典型错误）。

**这不是本次修复第一次踩这个坑**：commit message 与回归测试注释都明确提到，"2018 年该功能原型"就已经在同样的位置栽过。也就是说，即使有历史教训在前，"新增字符类时需要审计所有既有的按类判断函数"这条规则也没有被自动记住或写进任何检查清单——直到这次事故重演才被再次确认为通用规则。回归测试 `dashwidth01.lvt` 中专门有一条 `\TEST{em-dash after full stop}` 用例，测试用例的注释直接写明"2018 prototype of PoZheHao failed here with Missing number"，把这条教训显式固化到了可执行断言里。

**可推广的规则**：给字符分类体系新增一个类时，不能只关注"这个类本身该怎么处理"，必须反向搜索代码库里所有对旧类（本例是 `FullRight`）做**直接数值相等判断**（而非走字符类语义查询接口）的位置，逐一确认新类是否也该被这些判断接受。

## 关键发现 2：`\xeCJKResetPunctClass` 这类"重新声明整批字符类"的命令，与新增的专属子类之间存在隐式覆盖关系

`PoZheHao` 类的声明依赖用户开启 `PoZheHaoLigature` 选项；但 `\xeCJKResetPunctClass`（用户可调用的公开命令，用于重置标点符号所属字符类）内部会重新声明 `FullRight` 类的全部成员——这个重新声明动作会把 U+2014 重新拉回 `FullRight`，覆盖掉之前 `PoZheHaoLigature` 选项对它做的 `PoZheHao` 归类。

如果不处理这层关系，用户执行序列 "开启 `PoZheHaoLigature` → 调用 `\xeCJKResetPunctClass`（可能出于其他原因，比如自定义了别的标点分类后想重置）" 会静默丢失合字支持，且没有任何警告或错误——纯粹是"选项看起来生效了，实际排版结果却退化"的隐性失效。

修复方式是引入一个持久化的全局布尔 `\g_@@_pozhehao_ligature_bool` 记录用户的选项状态（而不是只让选项的 `.code:n` 一次性执行声明），`\xeCJKResetPunctClass` 执行完原有的重新声明逻辑后，在尾部检查这个布尔值，如果为真就重新执行一次 `PoZheHao` 类声明，把状态"找补"回来。

**可推广的规则**：当某个字符类的归属会被用户可调用的"重置/批量重新声明"类命令覆盖时，不能只在选项的 `.code:n` 里写一次性副作用，必须用持久状态变量记录"用户期望的最终状态"，并在所有可能重新声明基类的命令尾部检查并恢复。这类命令的识别特征是：函数体内出现 `\xeCJK_declare_char_class:nN` 或等价的批量声明调用，且该批量声明覆盖的字符集合与新增子类有交集。

## 关键发现 3：U+2014 与 U+2015 在 `PoZheHaoLigature=false` 时不对称恢复到同一个类，必须实测验证不能假设

`PoZheHaoLigature` 关闭时的恢复目标：U+2014 回到 `FullRight`，U+2015 回到 `Default`（普通类，编号 0）。如果凭直觉，容易假设两个字符对称处理（比如都恢复到 `FullRight`，因为二者都算某种破折号/连接号）。但实际上 U+2015（水平线）自 v3.3.3 起就已经不再属于 `FullRight` 类了——这是历史遗留的分类决策，与本次 `PoZheHao` 功能无关，但本次实现"关闭时的恢复逻辑"必须对齐这个历史现状而不是想当然地对称处理。

这一点在本次修复中被"实测验证"——即不是靠读代码注释确认，而是靠实际跑一遍开关往返（`PoZheHaoLigature` 开 → 关）后检查字符类归属是否与关闭前一致。回归测试 `dashwidth01.lvt` 中的 `\TEST{PoZheHaoLigature key toggles char class}` 用例覆盖了"开启合字 → 验证宽度 → 关闭合字 → 验证宽度恢复"的完整往返路径。

**可推广的规则**：任何"开关型选项，关闭时把状态恢复到某个基线"的实现，如果涉及多个字符/对象且它们的历史基线并不对称，必须对每一个对象单独核实其"关闭前的原始状态"，不能批量假设。往返测试（开→关→核对是否等于最初状态）是验证这类逻辑最直接的手段。

## 关键发现 4：标点度量的 cmap-blind 限制——诊断方法是"同一字体、开关某 OpenType 特性前后，测量结果是否变化"

xeCJK 所有标点尺寸都通过 `\fontcharwd` 与 `\XeTeXglyphbounds n \XeTeXcharglyph` 获取字符度量。`\XeTeXcharglyph` 本质是基于 cmap 的直接查表（字符 codepoint → 字形 ID），完全不经过 OpenType 的 GSUB/shaping 管线。这意味着当 `locl`（区域本地化替换）或 `fwid`（全角变体替换）等特性把原字形替换成另一个字形后，xeCJK 侧拿到的度量依然是**替换前那个字形**的度量——这是一个静态的、架构级的限制，不是某个函数的 bug。

**这个限制在本次修复复测过程中才被明确诊断出来**，具体表现为：裸 Noto Serif CJK SC（未显式在 `RawFeature` 里开启 `+fwid`）下，连用两个破折号的总宽经过公式修正后仍然是 1.78 ccwd，而不是 CLReq 要求的 2.0；但同一份文档、只改动 `\setCJKmainfont` 的 `RawFeature=+fwid` 参数后重新编译，总宽立刻变成 2.0（达标）。

**诊断方法的可推广性**：这个"字体不变、只切换某个 OpenType 特性开关，观察度量结果是否随之变化"的对比实验，本身就是判断"某条度量获取路径是否 feature-blind"最直接、成本最低的方法——不需要深入理解 XeTeX/HarfBuzz 内部 shaping 引擎的实现细节，只要能确认"同一份 `\fontcharwd`/`\XeTeXglyphbounds` 调用在开关特性前后返回值恒定"，就已经是"此路径不感知该特性"的充分证据。这与关键发现 3 里"往返测试验证选项恢复正确性"是同一类方法论——**用可观察的输出差异（或恒定）代替对内部实现的猜测**。

## 调试方法回顾：`\showbox` 对比法仍是这条子系统的核心分析工具

延续此前 #284（全角波浪号）反思里确立的方法论：本次修复的验证也大量依赖 `\showbox` 对比——开启 `PoZheHaoLigature` 且使用思源系字体后，`\showbox` dump 应该呈现"连续两个 U+2014 被合并为单一字形节点"（合字生效的直接证据），而非"两个独立字形节点 + 中间的 kern/glue"（合字未生效，退化为公式修正路径）。

这类问题（标点间距、字符合字）如果只看最终盒子总宽度是否达标，容易掩盖"节点结构是否真的走了预期路径"这一层信息——例如总宽凑巧达标但实际节点结构是"公式修正的近似值恰好等于 2 字宽"而非"合字触发的单一字形"，这两者在数值上可能巧合一致，但对后续维护（比如未来某个字体升级后合字条件变化）的影响完全不同。`\showbox` 是唯一能区分这两种情况的手段。

## 可复用教训汇总

- 新增字符类时，必须反向审计所有对旧类做直接数值判断（而非走语义接口）的函数，`\@@_punct_if_right:N` 这类判断函数是首要审计对象；历史上已经踩过的坑（如 2018 原型）如果没有被写进可执行的回归测试，会在功能重做时原样重演。
- 字符类归属如果依赖某个可开关选项，且系统里存在"批量重新声明字符类"的公开命令（如 `\xeCJKResetPunctClass`），必须用持久状态变量记录用户选项状态，并在这类命令尾部主动恢复，不能假设一次性声明能长期生效。
- 开关型选项的"关闭时恢复基线"逻辑，如果涉及多个字符对象，禁止假设对称恢复；必须对每个对象核实其原始历史分类（如 U+2015 自 v3.3.3 起不在 `FullRight`），并用开→关往返测试验证。
- 判断某条度量获取路径是否感知某个 OpenType 特性（是否 feature-blind），最直接的方法是保持字体不变、只切换该特性开关，对比测量结果是否变化——恒定不变即为"此路径不感知该特性"的证据，无需深入 shaping 引擎实现。
- `\showbox` 节点级 dump 对比仍是标点间距/合字类问题的核心分析工具，尤其要用它区分"总宽度巧合达标"与"真正走了预期的节点生成路径"这两种表面相似但本质不同的结果。
