# #553：可行性证明与产品化判断必须分开

## 任务

调查“ASCII 数字使用 CJK 字体但保持西文间距”的 feature request，识别真实需求，并以 MWE、节点列表和可见对比图说明结论。

## 先纠正问题模型，而不是沿用 feature 表述

表面请求是给数字增加一种字符类，真实需求却是“按字符范围选择另一字体，同时保留字符既有排版语义”。后者是 composite-font/range-to-font 路由问题；一旦这样重述，就会自然追问全局替换是否包含数学、页码、计数器、引用、URL、代码，以及字体族、字重和字形如何跟随。这些未定义作用域比数字类的实现细节更决定接口是否成立。

同样，用户所说“全部数字”不应直接当作授权去改所有上下文。先找出可显式标记的语义内容，可以用局部字体命令满足核心排版目标，并避开全局副作用。

## 不可能性判断需要最小反例

先前依据“每个字符只能属于一个 `\XeTeXcharclass`”断言字体与间距不能分离，混淆了“分类编号唯一”与“transition 能做什么”。最小原型复制 `Default` transitions，并在进入/离开新类时加入 CJK 字体分组，立即构成反例。

这类判断不能只读抽象限制。应先构造最小实现，逐项验证目标输出：字体节点是否真的改变、需要的 glue 是否存在、不需要的边界是否没有 glue。#553 中盒宽变化和 `\showbox` 字体名共同证明字形度量确实来自另一字体；截图负责解释视觉差异，节点列表负责排除“看起来差不多”的歧义。

## “能做”与“应做”是两次独立评审

原型只覆盖普通文本，正式类却会落入 xeCJK 的基本分类矛盾：字体语义属于 CJK，间距语义属于非 CJK。评估产品化时不能沿着原型代码量估算，而要反向搜索系统依赖的旧假设。

本次风险清单来自 `\g_@@_non_CJK_class_seq`、`\g_@@_CJK_class_seq` 及其消费者：标点、Boundary、math、listings、fntef、fallback、字体形状、颜色、链接和外部 class。#382 已证明新增字符类会遗漏对旧类的直接判断；#44 则提供复杂 spacing 状态长期维护成本的历史证据。两者都说明局部原型短小不代表接口影响面小。

## 可复用流程

处理 feature request 时依次完成四件事：把请求翻译成引擎无关的真实需求；用最小原型验证“可能/不可能”；用节点或状态证据验证原型而非只看截图；最后单独审计它会破坏哪些架构假设、需要哪些上下文语义和测试矩阵。

若已有低风险显式方案、全局作用域未定义、需求证据稀少且会破坏基本假设，应明确记录“技术可行但 not planned”。这比用错误的底层不可能性理由拒绝更准确，也给未来在证据变化时重新设计通用接口留下空间。
