# 反思：#986/#987 第三方宏包文档与验证强度

## 把组合方法与运行时集成分开

本次没有给 CTeX 增加 Babel 或 biblatex 依赖，也没有新增检测或补丁。手册只把已经
验证的组合方式写成可复制示例：Babel 保存语言状态，用户在公开语言钩子中调用
`\ctexset`；biblatex 管理 locale strings 与 heading，CTeX 不介入其内部生命周期。

因此，文档中的完整 MWE 和渲染图是用户指引，不应被架构文档误记为新的运行时集成。
Polyglossia 也只记录为独立的冲突边界，不能从 Babel 示例类推支持承诺。

## 自动审查建议仍要逐项核实

PR #988 第一轮建议不要覆盖 biblatex 内置的 `bibintoc`，改用独立 heading 名称后示例
意图更清楚；第二轮建议把默认 locale 为 `english` 的原因直接写入代码注释，也能帮助
只复制 MWE 的读者。这两项虽然标为非阻塞，仍有实际价值。

最后一轮关于 `[\bibname]` 的提醒则不成立为缺陷：biblatex 自带的 book 类 heading
正是以 `\bibname` 为默认标题，而且语言切换时会把它重新绑定到当前
`bibliography` locale string。`\chinese` 来源的注释也不会改善 CTeX 用户的用法。
审查闭环应当处理有效问题，并用上游定义或最小实验否决不成立的建议，而不是为了消除
所有提示继续堆叠说明。

## 验证强度应跟随当前增量

主体文档完成后运行一次 `l3build ctan` 很合适：184 项回归、手册和压缩包在同一入口
完成验证。随后只是替换 heading 名称和增加一行解释注释，重复全量运行没有增加相称的
置信度；第三次重复构建被中止后还留下了缺少临时日志的局部失败噪声。

更稳妥的规则是按本轮增量重新评估风险。已经有完整基线后，纯注释或措辞调整使用
`git diff --check`、必要的定向 MWE，并交给强制 CI 完成跨平台验证；只有改动触及运行时、
文档解析边界、示例语义、生成物或测试基线时，才再次升级到完整本地门禁。
